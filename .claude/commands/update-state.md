# /update-state — Refresh DB & Endpoints State Reports

You are a state auditor for Luna Legal AI. Your job is to produce (or refresh) two markdown reports in `agents_reports/`:

1. `db_state.md` — Current state of the Supabase database
2. `endpoints_state.md` — Current state of all backend API endpoints

## Step 1: Database State Report (`agents_reports/db_state.md`)

Query the Supabase project `dwgghvxogtwyaxmbgjod` to build a full snapshot of the database.

### 1a. Get table list with FULL column details

Use `mcp__supabase__list_tables` with `project_id: "dwgghvxogtwyaxmbgjod"`, `schemas: ["public"]`, `verbose: true` to get every table with all columns, types, defaults, PKs, FKs, and constraints.

### 1b. Get enum types

Run this SQL via `mcp__supabase__execute_sql`:
```sql
SELECT t.typname AS enum_name, string_agg(e.enumlabel, ', ' ORDER BY e.enumsortorder) AS values
FROM pg_type t
JOIN pg_enum e ON t.oid = e.enumtypid
JOIN pg_namespace n ON t.typnamespace = n.oid
WHERE n.nspname = 'public'
GROUP BY t.typname
ORDER BY t.typname;
```

### 1c. Get RLS status per table

Run this SQL via `mcp__supabase__execute_sql`:
```sql
SELECT tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;
```

### 1d. Get vector columns (embedding status)

Run this SQL via `mcp__supabase__execute_sql`:
```sql
SELECT c.table_name, c.column_name,
  (SELECT count(*) FROM information_schema.columns ic WHERE ic.table_name = c.table_name AND ic.table_schema = 'public') as total_cols
FROM information_schema.columns c
WHERE c.data_type = 'USER-DEFINED' AND c.udt_name = 'vector' AND c.table_schema = 'public'
ORDER BY c.table_name;
```

### 1e. Get embedding fill rates

For each table with a vector column, run:
```sql
SELECT 'TABLE_NAME' as table_name,
  count(*) as total_rows,
  count(embedding) as embedded_rows,
  round(100.0 * count(embedding) / GREATEST(count(*), 1), 1) as fill_pct
FROM TABLE_NAME;
```
(Replace TABLE_NAME for each table found in 1d)

### 1f. Get extensions

Use `mcp__supabase__list_extensions` with the project ID.

## Step 2: Endpoints State Report (`agents_reports/endpoints_state.md`)

Hit the live backend to check every endpoint's availability.

### 2a. Health check first

```bash
curl -s -o /dev/null -w "%{http_code}" https://luna-backend-production-35ba.up.railway.app/api/v1/health
```

If this returns non-200, note the backend is DOWN and skip live checks — instead produce the report from code analysis only.

### 2b. Check each endpoint group

For each endpoint, run a curl and record the HTTP status code. Use `-s -o /dev/null -w "%{http_code}"`.

**Auth endpoints** (no token needed for login/register):
- `POST /api/v1/auth/login` — send `{"email":"test@test.com","password":"test"}` (expect 401 = endpoint alive)
- `POST /api/v1/auth/register` — send `{"email":"probe@probe.com","password":"x","full_name_ar":"x"}` (expect 400 or 422 = alive)
- `POST /api/v1/auth/refresh` — empty body (expect 401 or 422 = alive)
- `GET /api/v1/auth/me` — no token (expect 401 or 403 = alive)
- `POST /api/v1/auth/logout` — no token (expect 401 or 403 = alive)

**Protected endpoints** (expect 401/403 without token = alive):
- `GET /api/v1/cases`
- `POST /api/v1/cases`
- `GET /api/v1/conversations`
- `POST /api/v1/conversations`
- `GET /api/v1/conversations/{uuid}/messages`
- `POST /api/v1/conversations/{uuid}/messages`
- `GET /api/v1/cases/{uuid}/documents`
- `POST /api/v1/cases/{uuid}/documents`
- `GET /api/v1/cases/{uuid}/memories`
- `POST /api/v1/cases/{uuid}/memories`
- `GET /api/v1/artifacts/conversation/{uuid}`
- `GET /api/v1/artifacts/case/{uuid}`
- `GET /api/v1/users/preferences`
- `PATCH /api/v1/users/preferences`
- `GET /api/v1/users/templates`
- `POST /api/v1/users/templates`

For protected endpoints, use a dummy UUID like `00000000-0000-0000-0000-000000000000`.

**Classify response codes:**
- 401/403 = ALIVE (auth guard working)
- 404 = ALIVE (route exists, resource not found)
- 422 = ALIVE (validation working)
- 200/201 = ALIVE
- 500 = ERROR (server error)
- 000/timeout = DOWN

## Rules

- Always run ALL checks — do not skip any even if some fail.
- Use the current date/time in the report headers.
- If a report already exists, overwrite it completely with fresh data.
- The RAG tables (entities, regulations, sections, articles, regulation_tables, regulation_images, cases, services) have NO RLS — they are the knowledge base. The rest are app tables WITH RLS.
- Do NOT modify any data. This is read-only.
- Run Supabase queries and curl checks in parallel where possible for speed.
- After writing both files, print a brief summary of findings to the user.

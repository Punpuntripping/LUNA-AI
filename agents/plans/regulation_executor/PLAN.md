# Regulation Executor -- Pydantic AI Implementation Plan

## What This Agent Does

The regulation executor is a non-interactive, single-turn agent that searches Saudi regulations, sections, and articles via hybrid retrieval (vector search + text fallback + Jina reranking). It is called as a tool by the deep search planner and returns a structured result containing ranked legal provisions with quality assessment, formatted markdown, and structured citations. It does NOT interact with users, produce artifacts, or manage task state.

## Agent Definition

| Setting | Value | Justification |
|---------|-------|---------------|
| Model | Configurable via `get_agent_model("search_regulations")` -- `gemini-3-flash` | Fast, cheap, tool-capable. The executor's job is retrieval orchestration, not deep reasoning. Gemini 3 Flash is 218 t/s at $0.50/$3.00 per 1M tokens. |
| output_type | `RegulationSearchResult` (structured BaseModel) | Planner needs quality signal, formatted markdown, and structured citations to decide whether to re-search. |
| deps_type | `RegulationSearchDeps` (separate from SearchDeps) | Needs `http_client`, `jina_api_key`, `supabase`, `embedding_fn` -- different shape than planner deps. |
| instructions | static only | No dynamic instructions needed. No case memory, no user interaction. |
| run method | `agent.run()` | Single-turn, no streaming, no iteration needed. |
| end_strategy | `early` (default) | Agent should return as soon as it has a result. |
| retries | 1 | Retry once on malformed output. |
| usage_limits | `UsageLimits(response_tokens_limit=4000, request_limit=5, tool_calls_limit=8)` | Executor should complete in 1-3 model requests. Safety net for edge cases. |

## Architecture

```
Deep Search Planner
    |
    | search_regulations(query="الفصل التعسفي نظام العمل")
    |
    v
┌──────────────────────────────────────────────────────┐
│              Regulation Executor Agent                │
│                                                      │
│  1. embed_and_search(query)                          │
│     ├── Generate embedding via gemini-embedding-001  │
│     ├── Vector search: articles (top 20)             │
│     ├── Vector search: sections (top 10)             │
│     └── Vector search: regulations (top 5)           │
│                                                      │
│  2. text_search_fallback(query)  [if vector weak]    │
│     └── pg_trgm ILIKE on articles.content            │
│                                                      │
│  3. rerank_results(query, candidates)                │
│     └── Jina Reranker API → top-K scored results     │
│                                                      │
│  4. unfold_context(top_results)                      │
│     ├── Load parent regulation metadata              │
│     ├── Load sibling articles (context window)       │
│     └── Truncate to budget                           │
│                                                      │
│  5. Return RegulationSearchResult                    │
│     ├── quality: "strong" | "moderate" | "weak"      │
│     ├── results_md: formatted markdown               │
│     └── citations: list[Citation]                    │
└──────────────────────────────────────────────────────┘
    |
    | RegulationSearchResult (structured)
    |
    v
Deep Search Planner (evaluates quality, decides next step)
```

### Data Flow

1. **Planner** calls `search_regulations(query)` tool on the planner agent
2. **Tool function** builds `RegulationSearchDeps`, runs `regulation_executor_agent.run(query, deps=reg_deps)`
3. **Executor agent** calls its own tools: `embed_and_search`, optionally `text_search_fallback`, `rerank_results`, `unfold_context`
4. **Executor agent** synthesizes findings into `RegulationSearchResult` structured output
5. **Tool function** serializes `RegulationSearchResult` to markdown string and returns it to the planner
6. **Planner** evaluates quality field, decides whether to re-search or synthesize

## Embedding Model

| Setting | Value |
|---------|-------|
| Model | `gemini-embedding-001` |
| Dimensions | 768 |
| API | Google AI `google.genai` client |

**Critical finding from DB investigation:** All existing embeddings in `regulations`, `sections`, and `articles` are **768 dimensions**, not the 1536 dimensions that `text-embedding-3-small` (OpenAI) would produce. The existing `agents/utils/embeddings.py` uses OpenAI's `text-embedding-3-small` which produces 1536-dim vectors. This means **the existing embeddings were NOT generated by the current `embed_text()` function** -- they were generated by a different model (likely Gemini) during the ingestion pipeline.

**Implication:** The regulation executor MUST use a 768-dim embedding model for vector search to match the stored embeddings. `gemini-embedding-001` produces 768-dim vectors by default and is the correct choice. The executor will have its **own** embedding function (`embed_regulation_query`) separate from the planner's OpenAI-based `embed_text`.

### New Embedding Utility

A new function `embed_regulation_query(text: str) -> list[float]` will be added to `agents/utils/embeddings.py` using the Google `genai` client. This function:
- Uses `gemini-embedding-001` model
- Returns 768-dim vectors
- Is used ONLY by the regulation executor (and future section/article executors)
- Coexists with the existing `embed_text()` which remains for app-side embeddings

## SearchDeps vs RegulationSearchDeps -- Tradeoff Analysis

### Option (a): Extend existing `SearchDeps`

Add `http_client: httpx.AsyncClient | None = None` and `jina_api_key: str = ""` to the existing `SearchDeps` dataclass.

**Advantages:**
- Single deps class across planner and executors -- simpler mental model
- Planner's `build_search_deps()` already runs in the orchestrator; extending it avoids a second factory
- When executor is called as a planner tool, deps pass-through is trivial (`ctx.deps`)

**Disadvantages:**
- Bloats `SearchDeps` with fields only one executor uses (`http_client`, `jina_api_key`)
- Violates single-responsibility -- planner deps should not know about reranker internals
- Every new executor (cases, compliance) would add more fields, causing combinatorial growth
- Testing the planner requires providing executor-specific fields (or accepting defaults)
- Tight coupling -- changing reranker config affects the planner's type signature

### Option (b): Separate `RegulationSearchDeps`

Create a new `RegulationSearchDeps` dataclass with only the fields the executor needs. The planner's `search_regulations` tool constructs it from `SearchDeps` + config.

**Advantages:**
- Clean separation of concerns -- each agent owns its own deps shape
- Executor is independently testable without planner machinery
- Adding new executors does not change the planner's deps
- Each executor's deps are minimal and self-documenting

**Disadvantages:**
- Two deps classes to maintain
- The `search_regulations` tool must construct `RegulationSearchDeps` from `SearchDeps` (a few lines of glue code)
- Slightly more complex import graph

### Recommendation: Option (b) -- Separate `RegulationSearchDeps`

The separation cost is minimal (one small factory function in the tool), while the benefits are substantial for long-term maintainability. As Luna adds `CaseSearchDeps`, `ComplianceSearchDeps`, etc., each executor stays self-contained. The planner never needs to know about reranker API keys or HTTP clients.

## Truncation Strategy

The executor needs truncation at two levels to prevent context window overflow:

### Level 1: Per-Result Truncation (unfold_context)

When a top-ranked article is unfolded with its parent regulation metadata and sibling articles, the combined text can be very large (max article is 20,519 chars). Each unfolded result is capped at:

| Field | Max chars | Rationale |
|-------|-----------|-----------|
| `content` (article/section text) | 3,000 chars | ~750 tokens. Covers 99% of articles (avg is 595 chars). Only truncates the rare monster articles. |
| `article_context` (AI-generated context) | 500 chars | Summary field, already concise. |
| `sibling_articles` (neighboring articles for context) | 1,500 chars total | 2-3 siblings at ~500 chars each. Enough for local context. |
| `regulation_metadata` (title, type, authority) | 300 chars | Fixed fields, rarely exceeds this. |

**Per-result cap: ~5,300 chars (~1,325 tokens)**

### Level 2: Total Budget Across All Results

| Setting | Value | Rationale |
|---------|-------|-----------|
| Max results returned | 10 | After reranking, keep top 10. |
| Total markdown budget | 40,000 chars (~10,000 tokens) | Fits comfortably in the executor's context window AND the planner's context window when the executor result is returned as a tool response. |

**How truncation works in practice:**
1. Reranker returns top-K results sorted by relevance score
2. `unfold_context` iterates from most relevant to least relevant
3. Each result is truncated to per-result caps
4. Results are accumulated until the total budget is reached
5. Remaining results are dropped with a note: "N additional results omitted due to budget"
6. If fewer than 3 results fit, the per-result cap is relaxed to 6,000 chars to prioritize depth over breadth

## Timeout Architecture

### Executor Agent Timeout (overall)

| Setting | Value | Rationale |
|---------|-------|-----------|
| Executor `agent.run()` timeout | 25 seconds | Includes embedding (1-2s) + 3 vector searches (1-2s each) + reranking (2-3s) + model reasoning (5-10s). 25s provides headroom for slow network. |

This is enforced by wrapping `agent.run()` in `asyncio.wait_for(coro, timeout=25)` inside the planner's `search_regulations` tool. If the executor times out, the tool returns an error string to the planner.

### Planner's `search_regulations` Tool Timeout

| Setting | Value | Rationale |
|---------|-------|-----------|
| `@planner_agent.tool(retries=1, timeout=30)` | 30 seconds | 25s for the executor + 5s buffer for deps construction and result serialization. Matches the deep_search plan's existing spec. |

### Individual Tool Timeouts (within executor)

| Tool | Timeout | Rationale |
|------|---------|-----------|
| `embed_and_search` | 10s | Embedding API call + 3 DB vector searches. |
| `text_search_fallback` | 5s | Single DB ILIKE query. |
| `rerank_results` | 8s | HTTP call to Jina API with up to 35 candidates. |
| `unfold_context` | 5s | DB lookups for parent metadata + siblings. |

## Scope Boundaries

### In Scope
- Vector search across `articles`, `sections`, and `regulations` tables
- Text search fallback via `pg_trgm` ILIKE when vector search yields weak results
- Jina reranking of combined candidates
- Context unfolding (parent regulation metadata, sibling articles)
- Quality self-assessment (strong/moderate/weak based on top reranker score + result count)
- Structured output with quality signal, markdown, and citations
- Per-result and total truncation to stay within token budget

### Out of Scope
- `regulation_tables` and `regulation_images` (skipped per user decision)
- User interaction (no ask_user, no respond_to_user)
- Artifact creation (planner's responsibility)
- Task state management (planner/orchestrator responsibility)
- Court cases, government services (separate executors)
- Embedding generation for DB ingestion (separate pipeline)

## Integration Points

### Planner Tool Wiring

The planner's `search_regulations` tool in `agents/deep_search/agent.py` will be updated from returning mock data to calling the real executor:

```python
@planner_agent.tool(retries=1, timeout=30)
async def search_regulations(ctx: RunContext[SearchDeps], query: str) -> str:
    from agents.deep_search.executors.regulation_executor import run_regulation_search
    result_str = await run_regulation_search(query, ctx.deps)
    ctx.deps._sse_events.append({
        "type": "status",
        "text": f"تم البحث في الأنظمة: {query[:80]}...",
    })
    return result_str
```

### Model Registry

The `"search_regulations"` key already exists in `agents/utils/agent_models.py` pointing to `"gemini-3-flash"`. No changes needed to the model registry. The executor uses `get_agent_model("search_regulations")` to obtain its model instance.
